CONTEXTO GENERAL DEL PROYECTO
================================================================================

Estás trabajando en el backend de KÜCHE, una plataforma web para fabricación de cocinas inteligentes. El sistema tiene DOS MÓDULOS PRINCIPALES que funcionan de manera INDEPENDIENTE:

MÓDULO 1: LEVANTAMIENTO RÁPIDO
- Es una estimación simplificada para captar leads y clientes potenciales
- Se completa en 5 pasos progresivos
- Genera presupuestos aproximados con rangos de precio
- Permite asignar el seguimiento a empleados

MÓDULO 2: COTIZADOR PRO  
- Es una cotización detallada y técnica con especificaciones completas
- Incluye desglose exhaustivo de materiales, herrajes y mano de obra
- Genera documentación técnica para el taller de fabricación
- Tiene 4 secciones principales: datos, escenarios, refinamiento y cierre

AMBOS MÓDULOS SON INDEPENDIENTES - no hay conversión automática entre ellos por ahora.

MODELO 2: Levantamiento
--------------------------------------------------------------------------------

Nombre del archivo: models/Levantamiento.js

Este es el modelo MÁS IMPORTANTE para el módulo de levantamiento rápido. Almacena toda la información capturada en los 5 pasos del proceso.

ESQUEMA ANIDADO 1: historialEstado

Este subesquema se usa dentro de Levantamiento para trackear cambios de estado.

Campos del subesquema:
- estado: Texto obligatorio que representa el nuevo estado
- fecha: Fecha y hora, por defecto la fecha actual del sistema
- usuario: Texto que indica quién hizo el cambio
- notas: Texto opcional con comentarios adicionales
- _id: false (no generar ID para subdocumentos)

CAMPOS PRINCIPALES DEL LEVANTAMIENTO:

PASO 1 - INFORMACIÓN DEL CLIENTE (objeto anidado llamado "cliente"):
- cliente.nombre: Texto obligatorio
- cliente.direccion: Texto obligatorio  
- cliente.telefono: Texto obligatorio

PASO 2 - GEOMETRÍA:
- metrosLineales: Número obligatorio, debe ser mayor o igual a 0

PASO 3 - NECESIDADES:
- requiereIsla: Booleano, por defecto false
- alacenasAltas: Booleano, por defecto false
- tipoCubierta: Enumeración obligatoria con opciones exactas "Granito Básico", "Cuarzo", "Piedra Sinterizada"

PASO 4 - SELECCIÓN DE ESCENARIO:
- escenarioSeleccionado: Enumeración obligatoria con opciones "esencial", "tendencia", "premium"

CAMPOS DE CÁLCULO (se calculan automáticamente):
- precioBase: Número que almacena el cálculo de metrosLineales multiplicado por 5000
- factorMaterial: Número decimal que representa el multiplicador del material
- multiplicadorEscenario: Número decimal que representa el multiplicador del escenario
- precioEstimado: Número que es el resultado del cálculo completo
- rangoMin: Número que representa el precio mínimo (precioEstimado multiplicado por 0.93)
- rangoMax: Número que representa el precio máximo (precioEstimado multiplicado por 1.08)

PASO 5 - ASIGNACIÓN Y SEGUIMIENTO:
- empleadoAsignado: Referencia ObjectId al modelo Usuario
- estado: Enumeración con opciones "pendiente", "en_revision", "contactado", "cotizado", "rechazado", "archivado". Por defecto "pendiente"
- historialEstados: Array del subesquema historialEstado definido arriba

METADATA ADICIONAL:
- notas: Texto opcional para comentarios generales
- convertidoACotizacion: Booleano por defecto false
- cotizacionId: Referencia ObjectId al modelo Cotizacion (si fue convertido)
- timestamps: Activar createdAt y updatedAt

MIDDLEWARE PRE-SAVE CRÍTICO:

Este modelo DEBE tener un middleware que se ejecute antes de guardar (pre save hook) para calcular automáticamente los precios. La lógica es:

CONSTANTES DE CÁLCULO:
- Precio base por metro lineal: 5000 pesos mexicanos

FACTORES DE MATERIAL (objeto JavaScript):
- "Granito Básico" tiene factor 1.0
- "Cuarzo" tiene factor 1.2
- "Piedra Sinterizada" tiene factor 1.5

MULTIPLICADORES DE ESCENARIO (objeto JavaScript):
- "esencial" tiene multiplicador 0.9
- "tendencia" tiene multiplicador 1.1
- "premium" tiene multiplicador 1.35

CÁLCULOS A REALIZAR EN EL MIDDLEWARE:
1. precioBase = metrosLineales multiplicado por 5000
2. factorMaterial = buscar en objeto factoresMaterial usando tipoCubierta, si no existe usar 1.0
3. multiplicadorEscenario = buscar en objeto multiplicadoresEscenario usando escenarioSeleccionado, si no existe usar 1.0
4. precioEstimado = precioBase multiplicado por factorMaterial multiplicado por multiplicadorEscenario
5. rangoMin = redondear (precioEstimado multiplicado por 0.93)
6. rangoMax = redondear (precioEstimado multiplicado por 1.08)

Asignar todos estos valores calculados a los campos correspondientes del documento.

Llamar a next() al final del middleware.

INSTRUCCIONES TÉCNICAS:
- Usar levantamientoSchema.pre('save', function(next) para el middleware
- Dentro del middleware, this se refiere al documento que se está guardando
- Usar Math.round para redondear rangoMin y rangoMax

MODELO 3: Cotizacion
--------------------------------------------------------------------------------

Nombre del archivo: models/Cotizacion.js

Este es el modelo MÁS COMPLEJO del sistema. Almacena cotizaciones detalladas con múltiples secciones.

ESQUEMA ANIDADO 1: historialEstado

Exactamente igual que en Levantamiento:
- estado: Texto
- fecha: Fecha por defecto Date.now
- usuario: Texto
- notas: Texto opcional
- _id: false

ESQUEMA ANIDADO 2: herrajeItem

Este subesquema representa cada herraje seleccionado en la cotización.

Campos:
- herrajeId: Referencia ObjectId al modelo CatalogoHerraje
- nombre: Texto con el nombre del herraje (copia para referencia)
- precioUnitario: Número con el precio por unidad (copia para referencia)
- enabled: Booleano que indica si está habilitado, por defecto false
- cantidad: Número entero mayor o igual a 0, por defecto 0
- _id: false

CAMPOS PRINCIPALES DE LA COTIZACIÓN:

SECCIÓN A - DATOS DEL PROYECTO:
- cliente: Texto obligatorio con nombre del cliente
- clienteRef: Referencia ObjectId opcional al modelo Cliente
- tipoProyecto: Enumeración obligatoria con opciones "Cocina", "Closet", "vestidor", "Mueble para el baño"
- ubicacion: Texto obligatorio (ciudad/estado)
- fechaInstalacion: Fecha obligatoria
- medidas: Objeto anidado obligatorio con:
  - medidas.largo: Número obligatorio mayor o igual a 0
  - medidas.alto: Número obligatorio mayor o igual a 0
  - medidas.fondo: Número obligatorio mayor o igual a 0
  - medidas.metrosLineales: Número obligatorio mayor o igual a 0

SECCIÓN B - NIVEL DE ACABADOS:
- escenarioSeleccionado: Enumeración con opciones "esencial", "tendencia", "premium"
- multiplicadorEscenario: Número decimal calculado (0.92, 1.05 o 1.18)

SECCIÓN C - MATERIALES:
- materialBase: Enumeración obligatoria con opciones "melamina", "mdf", "tech"
- materialBaseRef: Referencia ObjectId al modelo CatalogoMaterial
- precioMaterialPorMetro: Número con el precio por metro (se obtiene del catálogo)
- colorTextura: Enumeración obligatoria con opciones "Blanco Nieve", "Nogal Calido", "Gris Grafito", "Fresno Arena"
- grosorTablero: Enumeración con opciones "16", "19", por defecto "16"
- factorGrosor: Número decimal calculado (1.0 para 16mm, 1.08 para 19mm)

SECCIÓN C - HERRAJES:
- herrajes: Array del subesquema herrajeItem definido arriba

SECCIÓN C - MANO DE OBRA Y EXTRAS:
- manoDeObra: Número mayor o igual a 0, por defecto 0
- flete: Número mayor o igual a 0, por defecto 0
- instalacion: Número mayor o igual a 0, por defecto 0
- desinstalacion: Número mayor o igual a 0, por defecto 0

SECCIÓN D - CÁLCULOS FINALES (se calculan automáticamente):
- subtotalMateriales: Número resultado del cálculo de materiales
- subtotalHerrajes: Número suma de todos los herrajes habilitados
- subtotalManoObra: Número suma de mano de obra, flete, instalación y desinstalación
- precioFinal: Número suma de todos los subtotales

ESTADO Y SEGUIMIENTO:
- estado: Enumeración con opciones "borrador", "enviada", "aprobada", "en_produccion", "lista_instalacion", "instalada", "rechazada", "archivada". Por defecto "borrador"
- historialEstados: Array del subesquema historialEstado

METADATA ADICIONAL:
- empleadoAsignado: Referencia ObjectId al modelo Usuario
- notas: Texto opcional
- origenLevantamiento: Referencia ObjectId al modelo Levantamiento (si proviene de uno)
- timestamps: Activar createdAt y updatedAt

MIDDLEWARE PRE-SAVE CRÍTICO:

Este modelo DEBE tener un middleware que se ejecute antes de guardar para calcular automáticamente todos los precios. La lógica es:

CÁLCULO 1 - FACTOR DE GROSOR:
- Si grosorTablero es igual a "19" entonces factorGrosor = 1.08
- Si grosorTablero es igual a "16" o cualquier otro valor entonces factorGrosor = 1.0

CÁLCULO 2 - SUBTOTAL DE MATERIALES:
subtotalMateriales = medidas.metrosLineales multiplicado por precioMaterialPorMetro (o 0 si no existe) multiplicado por factorGrosor

CÁLCULO 3 - SUBTOTAL DE HERRAJES:
Recorrer el array herrajes usando un reduce:
- Para cada herraje en el array:
  - Si enabled es true: sumar (cantidad multiplicado por precioUnitario)
  - Si enabled es false: sumar 0
- El resultado final es subtotalHerrajes

CÁLCULO 4 - SUBTOTAL DE MANO DE OBRA:
subtotalManoObra = (manoDeObra o 0) más (flete o 0) más (instalacion o 0) más (desinstalacion o 0)

CÁLCULO 5 - PRECIO FINAL:
precioFinal = subtotalMateriales más subtotalHerrajes más subtotalManoObra

Asignar todos estos valores calculados a los campos correspondientes del documento.

Llamar a next() al final del middleware.

INSTRUCCIONES TÉCNICAS:
- Usar cotizacionSchema.pre('save', function(next) para el middleware
- Para el reduce de herrajes: this.herrajes.reduce((sum, item) => { ... }, 0)
- Usar operador OR (||) para manejar valores undefined o null: (this.manoDeObra || 0)

MÉTODO: listar (exports.listar)
--------------------------------------------------------------------------------

Este método lista levantamientos con filtros opcionales y paginación.

ENTRADA: req.query con filtros opcionales:
- estado (texto)
- empleadoAsignado (ID)
- fechaDesde (fecha ISO)
- fechaHasta (fecha ISO)
- page (número, por defecto 1)
- limit (número, por defecto 10)

PROCESO:

1. Crear objeto filtros vacío

2. Aplicar filtros si existen en query:
   - Si hay estado: filtros.estado = estado
   - Si hay empleadoAsignado: filtros.empleadoAsignado = empleadoAsignado
   - Si hay fechaDesde o fechaHasta:
     - Crear filtros.createdAt como objeto vacío
     - Si hay fechaDesde: filtros.createdAt.$gte = new Date(fechaDesde)
     - Si hay fechaHasta: filtros.createdAt.$lte = new Date(fechaHasta)

3. Calcular skip = (page - 1) multiplicado por limit

4. Ejecutar dos queries en paralelo con Promise.all:
   Query 1: Levantamiento.find(filtros)
            .populate('empleadoAsignado', 'nombre email')
            .sort({ createdAt: -1 })
            .skip(skip)
            .limit(parseInt(limit))
   Query 2: Levantamiento.countDocuments(filtros)

5. Retornar status 200 con:
   - success: true
   - data: array de levantamientos
   - pagination: objeto con:
     - total: el count total
     - page: parseInt(page)
     - pages: Math.ceil(total / limit)
     - limit: parseInt(limit)

MANEJO DE ERRORES:
Try-catch, si error retornar status 500 con success false y message "Error al listar levantamientos"

PARTE 6: FORMATO DE RESPUESTAS DEL BACKEND
================================================================================

TODAS las respuestas del backend deben seguir un formato consistente para facilitar el manejo en el frontend.

--------------------------------------------------------------------------------
RESPUESTA EXITOSA (status 200 o 201):
--------------------------------------------------------------------------------

Estructura del JSON:
{
  "success": true,
  "data": [objeto o array con los datos solicitados],
  "message": "Mensaje descriptivo opcional"
}

Para listados con paginación:
{
  "success": true,
  "data": [array de elementos],
  "pagination": {
    "total": número total de documentos,
    "page": página actual,
    "pages": total de páginas,
    "limit": elementos por página
  }
}


--------------------------------------------------------------------------------
RESPUESTA DE ERROR (status 400, 404, 500):
--------------------------------------------------------------------------------

Estructura del JSON:
{
  "success": false,
  "message": "Mensaje descriptivo del error",
  "error": "Detalle técnico del error (solo en development)"
}

CÓDIGOS DE STATUS A USAR:
- 200: Operación exitosa (GET, PUT, PATCH, DELETE)
- 201: Recurso creado exitosamente (POST)
- 400: Error de validación o datos incorrectos del cliente
- 404: Recurso no encontrado
- 500: Error interno del servidor
- 501: Funcionalidad no implementada

PARTE 14: NOTAS FINALES IMPORTANTES
================================================================================

1. PDFs NO SE GENERAN EN EL BACKEND
   - El frontend tiene botones para generar PDFs pero lo hace localmente
   - El backend solo guarda y retorna datos
   - No implementar endpoints para generación de PDFs

2. PRECIOS SON CONFIGURABLES
   - Todos los precios vienen de las colecciones CatalogoMaterial y CatalogoHerraje
   - Nunca hardcodear precios en controllers
   - Siempre buscar en catálogo antes de crear cotización

3. LEVANTAMIENTOS Y COTIZACIONES SON INDEPENDIENTES
   - No hay conversión automática por ahora
   - Endpoint convertirACotizacion retorna 501 (no implementado)
   - Esto se puede agregar después si se requiere

4. SISTEMA DE ESTADOS CON HISTORIAL
   - Cada cambio de estado debe quedar registrado en historialEstados
   - Nunca sobrescribir el historial, siempre hacer push
   - Incluir fecha, usuario y estado nuevo en cada entrada

5. CORS DEBE ESTAR HABILITADO
   - Frontend corre en puerto diferente (probablemente 3000)
   - Backend corre en puerto 3001
   - Sin CORS las peticiones serán bloqueadas

6. MANEJO DE ERRORES CONSISTENTE
   - Siempre usar try-catch en métodos async
   - Siempre hacer console.error del error completo
   - Retornar formato consistente con success: false

7. VALIDACIONES SON CRÍTICAS
   - Validar todos los inputs antes de guardar
   - Retornar mensajes de error descriptivos
   - Usar status 400 para errores de validación

8. MULTIPLICADORES Y FACTORES EXACTOS
   - Los valores de los multiplicadores son MUY ESPECÍFICOS
   - No redondear ni aproximar estos valores
   - Deben coincidir exactamente con los del frontend

9. POBLAR (POPULATE) REFERENCIAS
   - Usar populate en GETs para retornar datos relacionados
   - No poblar en POST/PATCH, solo retornar IDs
   - Seleccionar solo campos necesarios en populate

10. SEED LIMPIA Y RECREA DATOS
    - El script de seed ELIMINA todo antes de insertar
    - Solo ejecutar en desarrollo, nunca en producción
    - Útil para testing y desarrollo inicial
